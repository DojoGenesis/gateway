## What v0.1.0 Shipped

A working Go workspace with 7 modules (235 files, ~84K LOC), CI/CD, Docker, and a compiled binary. Extracted from the `dojo-genesis/go_backend` monolith. The framework runs, serves OpenAI-compatible endpoints, and passes all tests with race detection.

## What v0.1.0 Left Unfinished

Three items surfaced consistently across the retro and the health audit:

1. **Orchestration is not standalone.** The `orchestration/` module is a stub. The real engine is in `server/orchestration/`. This means orchestration can't be consumed without importing the server module and its 50+ transitive dependencies.

2. **server/ is 68% of the codebase.** At ~57K LOC, it's the monolith in a new directory. Sub-packages like `agent/`, `handlers/`, `database/`, `streaming/`, and `trace/` are tightly coupled to the server struct.

3. **No release automation.** There's no Makefile, no tagged release, no changelog, and no Goreleaser or equivalent. The "how to ship" process is manual.

## Candidate Focus Areas for v0.2.0

These are observations, not decisions. They're organized by the type of work involved.

### Extraction work (finishing what v0.1.0 started)

- Move `server/orchestration/` into standalone `orchestration/` module
- Evaluate whether `server/trace/` should become a standalone `trace/` module (it's used by orchestration, agent, and handlers — classic cross-cutting concern)
- Evaluate whether `server/streaming/` should merge into `events/` or remain server-internal
- Audit shared/types.go for types that leaked beyond the "currency types only" spec

### Developer experience work

- Add a Makefile (build, test, vet, lint, docker, clean)
- Add Goreleaser config for automated binary releases
- Generate OpenAPI/Swagger spec from the Gin routes
- Add Godoc comments to all exported types
- Create a CONTRIBUTING.md

### Quality work

- Add cross-module integration tests (provider → tools → orchestration pipeline)
- Add benchmarks for critical paths (SSE broadcasting, tool execution, memory compression)
- Set up test coverage thresholds in CI
- Lint with golangci-lint

### Product work

- OpenAI API conformance testing (validate against the actual OpenAI client libraries)
- Example plugins (a reference model provider plugin that third parties can copy)
- Example deployments (docker-compose, Kubernetes helm chart)

## Open Questions for Cruz

The retro produced clear action items, but the *priority* depends on where you want the project to go. Some framings to consider:

- **If the goal is open-source adoption:** Developer experience and documentation matter most. Makefile, examples, CONTRIBUTING.md, and a tagged v0.1.0 release.
- **If the goal is internal use in other Dojo products:** The orchestration extraction matters most. You want other services to import `orchestration/` without pulling in the server.
- **If the goal is production readiness:** Quality work matters most. Integration tests, benchmarks, lint, and OpenAPI spec.

These aren't mutually exclusive, but they suggest different orderings.
