# Retrospective: Agentic Gateway v0.1.0

**Date:** 2026-02-12
**Participants:** Cruz, Claude (Cowork)
**Context:** A reflection on the v0.1.0 release — extracting agentic infrastructure from the `dojo-genesis/go_backend` monolith into a standalone open-source framework. The work spanned context ingestion, strategic scouting, 4-track specification writing, pre-flight alignment, commissioning, and implementation.

---

## 1. The Three Core Questions

### What Went Well?

- **The spec-driven pipeline worked end-to-end.** From context ingestion of a 52K-line monolith through strategic scouting, spec writing, pre-flight checks, and commissioning — the full chain produced working code. The binary compiles, tests pass, CI runs.

- **Pre-flight caught real problems before they reached implementation.** The pre-flight report found 10 cross-spec inconsistencies (4 critical, 3 high, 3 medium) including divergent module paths across tracks, interface ownership conflicts, and stale Go versions. All were fixed before the implementation prompts were written. Without this step, the implementation agents would have built against contradictory specs.

- **The strategic scout prevented premature commitment.** Exploring 5 routes before committing to the hybrid approach (OpenAI-compatible + full agentic framework) surfaced trade-offs that a direct jump to spec writing would have missed. The name "Agentic Gateway by Dojo Genesis" emerged from the scout, not from brainstorming.

- **Parallel track structure kept specs manageable.** Four tracks (~45-53K characters each) were easier to reason about than a single monolithic spec would have been. Each track had clear boundaries and a defined dependency chain.

- **Module path standardization paid off.** Establishing `github.com/DojoGenesis/gateway/*` as canonical early meant the implementation agent had one source of truth. The late rename from `dojo-genesis/agentic-gateway` was a mechanical find-and-replace, not a design change.

### What Was Hard?

- **The orchestration extraction is incomplete.** The spec called for orchestration as a standalone module, but the implementation landed it as a stub with the real engine inside `server/orchestration/`. This means the "independently-usable modules" promise is only partially fulfilled — orchestration can't be consumed without importing server/.

- **server/ became a gravity well.** At 56,883 LOC (68% of the codebase), server/ absorbed functionality that the specs intended for standalone modules. The handlers, agent, orchestration, database, streaming, and trace subsystems all live there. This is the monolith pattern reasserting itself under a new name.

- **Two layers of handler wiring.** The server has both `handle_*.go` files at the root and a `handlers/` sub-package. This is legacy from the monolith extraction and creates confusion about where new endpoints should go.

- **Context window pressure across sessions.** The full pipeline (ingest → scout → spec → preflight → commission → implement) exceeded the context of any single session. The conversation required compaction and resumption, which meant re-establishing context and risking drift between what was decided and what was remembered.

- **Shared types scope drifted.** The spec said shared/ should be "currency types only" (Message, ToolCall, ToolResult, Usage, TaskStatus, NodeState). The implementation added CallRequest, CallResponse, ToolUse, ProviderConfig, and Timestamp — types that arguably belong in provider/ or are convenience wrappers. Shared/ grew from 6 types to 10+.

### What Would We Do Differently Next Time?

- **Add a "Track 0 verification" step after Track 0 implementation, before starting Tracks 1-3.** Track 0 is the foundation — if it drifts from spec (e.g., orchestration becoming a stub), that drift propagates. Verifying Track 0's output before commissioning dependent tracks would catch architectural shortcuts early.

- **Include a "module weight budget" in the spec.** State explicitly that server/ should not exceed X% of total LOC, and that any sub-package above Y LOC should be a candidate for extraction. This prevents the gravity well problem.

- **Write integration tests as a separate track.** The current test suite is mostly unit-level. A dedicated track for cross-module integration tests (e.g., "provider → tools → orchestration → server pipeline") would verify the module boundaries actually work as independent units.

- **Use a shared glossary document.** Terms like "provider", "plugin", "tool", and "event" had slightly different meanings across tracks. A single glossary referenced by all specs would have prevented the definitional drift.

## 2. Key Themes & Insights

| Theme | Analysis | Action Item |
| :--- | :--- | :--- |
| **Pre-flight alignment is non-negotiable** | 10 inconsistencies across 4 specs — including 4 critical architectural conflicts — would have caused implementation failure. The pre-flight step caught all of them. | Always run pre-flight checks on multi-track specs. Consider automating cross-spec consistency checks. |
| **Extraction gravity** | When extracting from a monolith, the largest module tends to absorb functionality that should be standalone. server/ became the new monolith. | Set explicit size budgets per module. Review module boundaries after implementation, not just during spec. |
| **Stub modules are technical debt** | The orchestration stub creates a false impression of modularity. The package exists, compiles, but doesn't work independently. | Either extract fully or don't create the standalone module at all. Stubs mislead consumers. |
| **Spec-to-implementation fidelity** | shared/types.go grew beyond spec scope. Not wrong per se, but it shows that implementation agents add "reasonable" types that weren't specified. | Include explicit "must contain" AND "must not contain" sections in specs for boundary-sensitive modules. |
| **The pipeline scales** | The full chain — ingest, scout, spec, preflight, commission, implement — produced a working 83K-LOC codebase from a monolith. The process is repeatable. | Document the pipeline as a reusable workflow. It's a pattern, not a one-off. |

## 3. Seeds for the Memory Garden

- **Seed:** Pre-Flight as Quality Gate — *Why it matters:* Multi-track specs diverge. The pre-flight step is where you catch it. Without it, implementation agents build against contradictory contracts. — *Revisit trigger:* Any time you're commissioning 2+ parallel specs.

- **Seed:** Module Weight Budget — *Why it matters:* Extraction from a monolith produces a new monolith unless you constrain module size. server/ at 68% of total LOC is the monolith wearing new clothes. — *Revisit trigger:* Any extraction or modularization project. Set the budget before spec writing, not after.

- **Seed:** Stub Modules Are Lies — *Why it matters:* A package that compiles but doesn't work independently creates false confidence. Either extract fully or mark the package as "internal to server" until it can stand alone. — *Revisit trigger:* Any time a module is "planned for future extraction."

---

## 4. Closing

v0.1.0 is a working product. It compiles, it passes tests, it has CI, it has Docker, it has documentation. The spec-driven pipeline proved itself as a process — from understanding a 52K-line monolith to shipping a modular framework in a structured sequence of steps. The main lesson: the process works, but extraction is harder than creation. The monolith always wants to come back. v0.2.0 should focus on finishing what v0.1.0 started — making the modules genuinely independent.
