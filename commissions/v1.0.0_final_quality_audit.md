## 2. Audit Methodology

Work through each audit pass sequentially. For each pass, report what you found and what you fixed. If a pass is clean, say so explicitly. Do not skip passes.

---

### Pass 1: Build & Test Verification

**Goal:** Confirm the codebase compiles and all tests pass.

1. Run `go build ./...` from the workspace root. It must succeed with zero errors.
2. Run `go vet ./...`. It must succeed with zero warnings.
3. Run `go test -race -count=1 ./...`. All tests must pass. Record the total test count.
4. If any test fails, diagnose and fix the root cause. Do not skip or disable tests.
5. Run `go test -race -coverprofile=coverage.out ./...` then filter protobuf: `grep -v 'provider/pb/' coverage.out > coverage_filtered.out` then `go tool cover -func=coverage_filtered.out | tail -1`. Record the aggregate coverage percentage.

---

### Pass 2: Version String Consistency

**Goal:** Every version string in the codebase must be `1.0.0` or be a justified exception.

1. Search the entire codebase for any occurrence of `0.1.0`, `0.2.0`, `0.2.4`, `0.3.0` in Go source files (exclude `CHANGELOG.md`, `docs/`, `go.sum`, `go.mod`, `.git/`, `vendor/`, `dist/`).
2. For each hit, determine:
   - **Stale:** The string refers to *this project's* version and should be `1.0.0`. Fix it.
   - **Correct:** The string refers to an external protocol version, schema version, or historical documentation. Leave it and note why.
3. Search for the string `"v0.` in Go source files (same exclusions). Apply the same stale-vs-correct analysis.
4. Verify these specific locations are `1.0.0`:
   - `server/server.go` — `const Version`
   - `server/handlers/health.go` — `Version` field in health response
   - `provider/manager.go` — `HandshakeValue`
   - `skill/adapters/web_tools.go` — User-Agent string

---

### Pass 3: Stale Comment & TODO Audit

**Goal:** No stale development-phase comments remain in production code.

1. Search all `.go` files (excluding `*_test.go`, `docs/`, `dist/`) for:
   - `Phase 4a` or `Phase 4b` (case insensitive)
   - `Phase 3` or `Phase 2` or `Phase 1` (only flag if they describe *current or future* work — historical provenance comments in `ARCHITECTURE.md` are fine)
   - `TODO` or `FIXME` or `HACK` or `XXX`
   - `placeholder` or `stub` or `not yet implemented` or `not implemented`
2. For each hit, categorize:
   - **Must fix:** The comment is stale, misleading, or references a development phase as if it's current. Rewrite to describe the actual current state.
   - **Acceptable:** The comment describes genuinely deferred work (post-v1) or is a legitimate code TODO with a clear scope. Leave it but record it.
   - **Test-only:** The comment is in a test file. Leave it — tests can reference phases for historical context.
3. Produce a final inventory: "N stale comments fixed, M acceptable TODOs remaining, K test-only."

---

### Pass 4: Dead Code & Unused Imports

**Goal:** No dead code ships in v1.0.0.

1. Run `go vet ./...` — it catches some unused variables and imports.
2. Search for functions or methods that are unexported (`func lowercase...`) and never referenced outside their own file. These may be dead code. Use judgment: helpers used only within a file are fine, but functions that were once called from deleted code should be removed.
3. Check for any `.go` files that exist but are not part of any build tag or package (orphaned files).
4. Check for any unused struct fields or interface methods that were added speculatively and never implemented.
5. If you find dead code, remove it. If removal is risky (uncertain whether something is used at runtime via reflection or plugin loading), add a `// TODO(post-v1): verify if this is still needed` comment instead.

---

### Pass 5: Configuration & Deployment Consistency

**Goal:** All configuration files, Docker, and CI are internally consistent and reference the correct version.

1. Read `gateway-config.yaml`. Verify:
   - All 8 providers are represented (Anthropic, OpenAI, Google/Gemini, Groq, Mistral, Kimi, DeepSeek, Ollama)
   - MCP server entries include SSE and streamable_http transport examples if the config supports them
   - No references to removed features (e.g., `embedded-qwen3`)
2. Read `docker-compose.yaml` and `docker-compose.example.yml`. Verify:
   - Gateway image tag is consistent (`:latest` or `:v1.0.0` as appropriate)
   - No stale environment variable references
   - Health check endpoints match the actual health handler
3. Read `Dockerfile` and `Dockerfile.goreleaser`. Verify:
   - Both build correctly structured images
   - No stale version references in labels or comments
4. Read `.github/workflows/ci.yml`. Verify:
   - All jobs are correctly configured
   - E2E smoke test validates the right endpoints
   - No references to removed binaries or test commands
5. Read `.goreleaser.yml`. Verify:
   - Per-arch Docker images configured for amd64 and arm64
   - Manifest lists configured for `:v{{ .Version }}` and `:latest`
   - No references to old version patterns

### Pass 6: Documentation Cross-Reference

**Goal:** Every claim in STATUS.md, ARCHITECTURE.md, README.md, and CHANGELOG.md is true and verifiable.

1. **STATUS.md — Aggregate Stats Verification:**
   - Count Go files: `find . -name '*.go' -not -path './.git/*' -not -path './dist/*' -not -path './vendor/*' | wc -l`. Must match the claim.
   - Count test files: `find . -name '*_test.go' -not -path './.git/*' -not -path './dist/*' | wc -l`. Must match.
   - Count API routes: read `server/router.go` and count registered routes. Must match the "37 endpoints" claim.
   - Count providers: `ls server/services/providers/*.go | grep -v _test | grep -v base | grep -v openai_compat | wc -l`. Must match "8 providers."
   - Count MCP transports: read `mcp/connection.go` and verify stdio, SSE, and streamable_http are all implemented.
   - Count migrations: `ls server/migrations/*.sql | wc -l`. Must match "25 migrations."
   - Count skills: verify the "44 skills" claim against actual plugin directories.
   - Count tools: verify the "33 tools" claim against the tool registry.
2. **ARCHITECTURE.md:**
   - Verify the directory structure diagram matches the actual filesystem.
   - Verify the component descriptions match what the code actually does.
   - Verify the version history section is complete (v0.1.0 through v1.0.0).
3. **README.md:**
   - Verify all quickstart commands work (build, run, docker).
   - Verify API examples reference correct endpoints and payloads.
   - Verify feature list matches actual capabilities.
   - Check all links (internal and external) for correctness.
4. For every discrepancy: fix the documentation to match the code. Code is truth; documentation follows.

### Pass 7: Import Hygiene & Module Consistency

**Goal:** The Go workspace is clean and dependency graph is correct.

1. Read `go.work` and verify all listed modules exist on disk.
2. For each module (`go.mod`), run `go mod tidy` and check if it produces any changes. If it does, the dependency graph was dirty — commit the tidied version.
3. Verify no module imports a sibling module at a stale version (check `go.work.sum` for red flags).
4. Check that `go.sum` files don't contain entries for removed dependencies.

---

### Pass 8: API Contract Verification

**Goal:** The public API surface is consistent and correctly documented.

1. Read `server/router.go` and inventory every registered route.
2. For each route:
   - Verify the handler function exists and is correctly wired.
   - Verify the HTTP method matches the handler's implementation.
3. Verify that the health endpoint returns `version: "1.0.0"` by reading the handler code.
4. Verify the `/admin/providers` endpoint returns live status for all 8 providers.
5. Check for any routes that are registered but point to stub/unimplemented handlers (returning 501). List them.

---

### Pass 9: Provider Layer Verification

**Goal:** All 8 providers are correctly implemented and tested.

1. For each provider in `server/services/providers/`:
   - Verify it implements the required interface.
   - Verify it has a corresponding test (either in `conformance_test.go` or its own `*_test.go`).
   - Verify the model list is reasonable and up-to-date for that provider.
2. Verify `RegisterProviders()` discovers all 8 providers at startup.
3. Verify the provider routing preferences in the router are correct and don't reference removed providers.

---

### Pass 10: Security & Sensitive Data Scan

**Goal:** No secrets, credentials, or sensitive data in the codebase.

1. Search for patterns that look like API keys: strings matching `sk-`, `key-`, `gsk_`, `AIza`, or any 32+ character hex/base64 strings in Go source or config files.
2. Verify `.gitignore` covers: `.env`, `*.pem`, `*.key`, `coverage.out`, `dist/`, `bin/`, `vendor/`.
3. Verify no test files contain hardcoded real API keys (mock keys like `test-key-123` are fine).
4. Verify `gateway-config.yaml` uses environment variable references (`${ANTHROPIC_API_KEY}`) rather than inline secrets.
